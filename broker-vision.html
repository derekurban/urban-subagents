Scope

What we are building

A management layer for sub-agents that delegates to existing CLIs instead of replacing them. The broker should expose one stable tool surface to the parent process while mapping that surface onto Claude Code or Codex underneath.

Responsibilities

  • Accept delegation requests from the parent process
  • Select a configured agent definition
  • Translate the request into Claude Code or Codex arguments
  • Track session ids, status, output, and resumable state
  • Return normalized results to the parent process

Non-goals

  • No direct vendor API harness
  • No reimplementation of auth, sandboxing, or session storage
  • No requirement to expose raw model names to the parent
  • No baked-in profile catalog in this document
The point is to broker delegation, not to become another agent runtime.
Interaction model

How the broker fits into the flow

Parent process
    |
    | calls broker tool
    v
Sub-agent broker
    |
    | resolves configured agent definition
    | applies policy and runtime mapping
    v
Provider adapter
    |
    +--> Claude Code adapter  -> claude -p / resume / MCP config / prompt flags
    |
    +--> Codex adapter        -> codex exec --json or app-server thread lifecycle
    |
    v
Normalized session registry
    |
    +--> result summary
    +--> structured output
    +--> status / cancel hooks
The critical split is between one-shot execution and resumable sessions. One-shot execution still matters, but both products can also emit session or thread handles that make resume possible. The broker should capture those handles even when it runs inline.
Delegation API

Minimal external command surface

The broker should keep the parent-facing API small. The semantics matter more than the exact names, but this is a workable v1 surface.

list_agents

Return configured agent definitions that the parent is allowed to delegate to.

  • Purpose: discovery
  • Typical output: name, description, runtime, write/network policy, resumable support

list_sessions

Return broker-known sessions for the current broker session or namespace.

  • Purpose: inspect current and recently used resumable handles
  • Typical output: session id, agent, runtime, provider handle, last status, created at, updated at
  • Recommended default scope: the current broker session, not every session on the machine

delegate

Run an agent definition either as a new session or by resuming an existing session.

  • Purpose: start new or resume by session_id
  • Typical input: agent name, prompt text, optional session_id, cwd, context, constraints
  • Typical output: session id, status, runtime metadata, provider handle, result

cancel (optional)

Request interruption for an in-flight session when the runtime or wrapper can stop it.

  • Purpose: stop an in-flight run early
  • Typical input: session id, optional reason
  • Typical output: cancellation requested state for that session
{
  "tool": "delegate",
  "agent": "reviewer",
  "prompt": "Review the auth patch for race conditions and session bugs.",
  "cwd": "D:/repo/app",
  "context": {
    "files": ["src/auth/session.ts", "src/auth/token.ts"],
    "notes": "Read-only. Findings first."
  }
}
{
  "tool": "delegate",
  "agent": "reviewer",
  "session_id": "sess_018",
  "prompt": "Narrow this to exploitable issues only."
}
{
  "tool": "list_sessions"
}
{
  "tool": "cancel",
  "session_id": "sess_018",
  "reason": "No longer needed"
}
This is a sync-first API. In the normal case, delegate runs inline and returns when the CLI run finishes. That means a separate wait tool is not part of the core surface. If detached execution is added later, wait can be added as an extension without changing the session model.
Lifecycle semantics

What a session means in this broker

The broker should normalize around sessions, not tasks. Each delegate call either creates a fresh session or resumes an existing one by session_id.

Recommended session states

  • running
  • idle
  • completed
  • failed
  • interrupted

Recommended invariants

  • No separate followup tool; resume is another delegate call
  • No core wait tool in sync-first mode
  • delegate without session_id creates a fresh session
  • delegate with session_id resumes the same logical conversation

Inline execution

The default mode. delegate blocks until the underlying CLI run exits and returns the result together with the broker session id and provider handle.

cancel

Useful only while a run is in flight. In sync mode this usually means the broker or host terminates the underlying subprocess or invokes a documented interrupt primitive.

Resume

A new delegate call with the same session_id. This is how “follow-up” works without requiring another top-level tool.

One-shot runtime policy

  • Examples: claude -p, codex exec --json
  • The current run finishes when the subprocess exits
  • The broker still captures any session or thread handle emitted by the backend
  • cancel means terminate the subprocess and update session state locally
  • Cleanup is simple: do not keep idle child processes alive after completion

Resumable runtime policy

  • Examples: Claude session resume flow, Codex app-server threads
  • The logical session can outlive a single process invocation
  • cancel should map to the runtime’s interruption primitive where one exists
  • Resume means another delegate call against the same session id
  • Cleanup becomes a retention question: keep the resumable handle for later or expire it after a configured window
Default recommendation: keep the API synchronous, capture backend session handles, and avoid background workers until there is a real need for detached execution.
Provider mapping notes

How the broker API maps onto Claude Code and Codex

Claude Code adapter

  • delegate maps naturally to claude -p with --output-format json
  • Model and effort map to --model and --effort
  • Prompt injection maps to --system-prompt-file or --append-system-prompt-file
  • Tool exposure maps to --tools, --allowedTools, and --disallowedTools
  • Broker-hosted tools map through --mcp-config
  • Official docs support resume through --continue or --resume <session_id> for claude -p runs
  • That means the broker can capture the returned session id and treat the next resume as another delegate call
  • Official docs do not describe an in-band cancel primitive for headless claude -p; broker cancel should therefore terminate the subprocess and mark the session locally
  • Claude's tool docs do document TaskStop for killing a running background task by ID, but that belongs to Claude's native task system rather than the broker contract
  • Native Agent can be denied so the parent uses the broker instead

Codex adapter

  • delegate maps cleanly to codex exec --json for one-shot tasks
  • Model selection maps to --model or config profile resolution
  • Reasoning maps to config overrides like model_reasoning_effort
  • Developer instructions can map to config-level instruction fields
  • Official docs support non-interactive continuation via codex exec resume --last or codex exec resume <SESSION_ID>
  • That means the broker can treat resume as another delegate call with the same session id
  • App-server gives thread lifecycle via thread/start, thread/resume, thread/fork, and turn/start
  • App-server documents a real interruption primitive: turn/interrupt, which ends the turn as interrupted
  • Dynamic tools and approvals are available if the broker later needs a richer runtime contract
Practical rule: use the simplest adapter that can satisfy the requested lifecycle. Use inline CLI execution by default. Capture the provider session handle every time. Only move to a persistent session backend when inline resume or interruption stops being sufficient.
Config shape

Example configuration for the delegation layer

This is not a profile catalog. It is the contract for describing available delegated agents and the runtimes they map to. 

version: 0.1

broker:
  scope_mode: session
  retain_sessions_for: 24h
  execution_mode: sync
  default_output:
    format: text

runtimes:
  claude_code:
    command: claude
    launch:
      mode: print
      base_args: ["-p", "--output-format", "json"]
    supports:
      resume: true
      interrupt: process_terminate
      write_control: true

  codex_exec:
    command: codex
    launch:
      mode: exec
      base_args: ["exec", "--json"]
    supports:
      resume: true
      interrupt: process_terminate
      write_control: true

  codex_app_server:
    command: codex
    launch:
      mode: app_server
    supports:
      resume: true
      interrupt: turn_interrupt
      write_control: true

agents:
  reviewer:
    description: Review code for bugs and regressions
    runtime: codex_exec
    model: gpt-5.4
    reasoning_effort: high
    prompt_file: prompts/reviewer.md
    permissions:
      writes: false
      network: false
    cwd_policy: explicit
    session_policy: new_or_resume
    output:
      format: text

  planner:
    description: Build implementation or migration plans
    runtime: claude_code
    model: opus
    effort: high
    prompt_file: prompts/planner.md
    permissions:
      writes: false
      network: false
    cwd_policy: explicit
    session_policy: new_or_resume
    output:
      format: json_schema
      schema_file: schemas/plan-result.json

  long_review:
    description: Resumable review session
    runtime: codex_app_server
    model: gpt-5.4
    reasoning_effort: high
    prompt_file: prompts/reviewer.md
    permissions:
      writes: false
      network: false
    session_policy: resumable
    output:
      format: text
broker.scope_mode Recommended default is session scope. The broker should list sessions for the current broker session or namespace, not every session on the machine.
broker.retain_sessions_for Controls how long finished session metadata remains available for inspection, resume, or debugging after the child process exits.
broker.execution_mode Recommended default is sync. In this mode delegate runs inline and returns when the CLI run finishes.
runtimes.* Defines the backend launch method. This is where provider-specific mechanics live.
agents.* Defines a delegatable worker. This is what the parent process should see and choose from.
model / effort / reasoning_effort These stay in config and get translated by the adapter. They do not need to be exposed in the normal parent API.
prompt_file The broker owns prompt packs and maps them to each provider's instruction surface.
session_policy Controls whether an agent definition always starts fresh, always expects resume, or allows both new and resumed sessions.
permissions Declares writes and network policy at the agent-definition level so the broker can enforce them before launch.
output Default recommendation is plain text because it is easiest for the orchestrator to read and reason over. Use a JSON schema only when downstream automation needs stable fields.
Open questions

What still needs to be decided

Session registry shape

Should list_sessions expose only broker session ids, or should it also surface provider session/thread ids for debugging?

Cancel scope

Should cancel be part of the core API from day one, or only added once detached or long-running execution proves necessary?

Output contract strictness

How often will plain text be enough, and when is JSON schema worth the extra coordination cost?
